<Section id="sec-1" title="Introduction">
<p>This wiki serves as the formal reference document for our family's ClickUp workspace. Its purpose
  is to explain, in a clear and systematic way, how the workspace is structured, what it is designed to support, and
  how its underlying logic should be understood and maintained over time.</p>
<p>The workspace itself is <LinkTarget as="span" id ="test">intended to help us bring greater order</LinkTarget>, visibility, and continuity to everyday life. It
  provides a shared system for organizing responsibilities, tracking tasks, planning events, capturing ideas, and
  preserving information that should not remain scattered across conversations, memory, or temporary notes. In that
  sense, it is not only a task manager, but a broader operational and documentation environment: a place where action
  items can be managed, commitments can be made visible, plans can be coordinated, and important knowledge can be
  recorded for future use.</p>
<p>This wiki exists to make that environment legible. As the workspace grows in complexity, a written reference becomes
  necessary in order to define concepts consistently, document design decisions, and ensure that the system remains
  understandable, intentional, and maintainable.</p>
<p>The wiki is organized into three main sections.</p>
<p>The <Link to="sec-2">Ontology</Link> section establishes the conceptual foundation. It defines the general vocabulary
  used to reason about structured information systems and databases, independently of any specific software platform.
  Its role is to clarify the broader concepts that underlie the rest of the document.</p>
<p>The <Link to="sec-3">ClickUp</Link> section explains how those abstract concepts are realized within ClickUp itself.
  It describes how ClickUp represents data, structure, fields, scopes, and behaviors, and how the platform's own
  implementation maps onto the conceptual framework introduced in the Ontology section.</p>
<p>The <Link to="sec-4">Workspace</Link> section documents our specific customization of ClickUp. It explains how we have
  adapted the platform to our own needs, practices, and use cases: which structures we rely on, how we organize
  information, what conventions we follow, and how the workspace is configured to support the way our family plans,
  manages, and records the different parts of life.</p>
<p>Together, these sections move from general concepts, to platform implementation, to our concrete usage of the
  platform. This progression is intentional: first defining the underlying ideas, then examining how ClickUp embodies
  them, and finally describing how we apply and tailor them in <Link to="test">practice</Link>.</p>

</Section>
<Section id="sec-2" title="Ontology">
<p>This section defines a precise vocabulary for reasoning about databases and no-code database platforms. It
  formalizes core concepts, and provides a conceptual foundation for analyzing how these platforms represent data,
  enforce structure, and apply behavior.</p>
<Section id="sec-2-1" title="Field">
<p>A reusable property descriptor: it defines a named "slot" that can hold a value.
  A field definition typically includes:</p>
<ul>
  <li><strong>Identity</strong>: stable field identifier</li>
  <li><strong>Name</strong>: human-readable presentation</li>
  <li><strong>Field type</strong>: what kind of data this field represents</li>
  <li><strong>Nullability:</strong> whether a value must be set</li>
  <li><strong>Default value</strong>: what value is assumed when none is provided</li>
  <li><strong>Scope</strong>: under what contexts it appears</li>
  <li><strong>Permissions</strong>: who can view/edit</li>
</ul>
</Section>
<Section id="sec-2-2" title="Schema">
<p>A specification that defines a valid entry. It consists of:</p>
<ul>
  <li><strong>Fields</strong>: the set of fields that apply.</li>
  <li><strong>Constraints</strong>: rules that determine validity. They may restrict the value of a single field, such
    as whether it is required, typed, or limited to a permitted range or set of values, and they may also govern the
    relationship between multiple fields, such as dependency, consistency, exclusivity, or conditional validity.</li>
  <li>
    <p>
      <strong>Operational rules</strong>: rules that define how field values change over time in response to
      defined triggers. Each operational rule includes:
    </p>
    <ul>
      <li>
        a <strong>trigger condition</strong>: specifies the circumstance under which the rule is activated. This may be
        the creation of a new entry under the schema, a change in a field value, or the passage of time.
      </li>
      <li>a <strong>state update</strong>: one or more field values are changed.</li>
    </ul>
    <p>Operational rules often exist to enforce constraints, although not necessarily.</p>
  </li>
</ul>
</Section>
<Section id="sec-2-3" title="Entry">
<p>A concrete instance governed by a schema. It consists of a set of values, one per
  field included in the schema.
  The set of an entry's field values at a given moment is its state.
  An entry is valid under its schema if each value belongs to its field's allowed domain and all
  schema constraints are satisfied.</p>
</Section>
<Section id="sec-2-4" title="Database">
<p>A  structured collection of entries governed by one or more schemas, together with the organizational framework that stores and makes those entries accessible..</p>
</Section>
<Section id="sec-2-5" title="Scope">
<p>A context layer that contributes a bundle of applicability for entries
  exposed to it. A scope can contribute:</p>
<ul>
  <li><strong>Schemas</strong> that apply in that context</li>
  <li><strong>Settings/policies</strong> (permissions, presentation rules, workflow behaviors)</li>
  <li><strong>Per-scope parameterization</strong> of fields (where the same field kind has different parameters in
    different contexts)</li>
  <li><strong>Automations</strong> that are triggered by conditions inside the scope.</li>
</ul>
<p>All of an entry's attributes are derived from the scopes it is exposed to, taken together: each scope contributes part of what applies to that entry, and the result is the union of those contributions.</p>
</Section>
<Section id="sec-2-6" title="Unified Database Model">
<p>In many no-code database platforms, the workspace often looks like it contains many separate databases scattered across locations, sections, or modules. Conceptually, though, you can model the entire workspace as one big database of entries with a shared universe of fields—where differences in what you see and what applies are explained by scopes.</p>
<Section id="sec-2-6-1" title="Scope Exposure">
<p>Entries become exposed to scopes through scope selector fields: fields whose values attach the
  entry to additional scopes. These are not merely data holders; they are context-binding mechanisms that determine which schemas/settings/rules apply.</p>
</Section>

<Section id="sec-2-6-3" title="Per-Scope Parameterization">
<p>Some field types are parameterized: their full meaning requires configuration parameters.
  Scopes can provide those parameters at different levels.
  For example:</p>
<ul>
  <li>a "status" field is one field kind, but the allowed state set or workflow graph can be configured per scope
    level</li>
  <li>a "choice" field can have different option sets in different contexts</li>
  <li>a templated or AI-driven field can have different prompts/specs per scope
    This allows consistent field kinds while enabling contextual variation.</li>
</ul>
</Section>
<Section id="sec-2-6-4" title="Scoped Scope Hierarchies">
<p>Scope hierarchies do not have to be globally accessible. You can define a scope selector field inside a
  particular scope so that only entries already exposed to that scope can access that selector, and only then can
  they branch into a further scope hierarchy.
  This yields nested or conditional modeling: certain context systems exist only inside certain other contexts.</p>
<p>For example, the Status Scope Hierarchy is scoped under the Task scope of the Type Scope Hierarchy. That scope exposes a status field that acts as a scope selector into the Status hierarchy, allowing entries within the scope to branch into a more specific context that models their progression state:</p>
<div class="doc-tree">
  <ul>
    <li>
      ⬜ Not Started
      <ul>
        <li>🟫 Backlog</li>
        <li>🟨 Todo</li>
      </ul>
    </li>
    <li>
      ⬜ Active
      <ul>
        <li>🟪 In Progress</li>
        <li>🟧 Blocked</li>
      </ul>
    </li>
    <li>
      ⬜ Done
      <ul>
        <li>🟥 Canceled</li>
        <li>🟩 Completed</li>
      </ul>
    </li>
  </ul>
</div>
</Section>
<Section id="sec-2-6-5" title="Entry-Local Scope">
<p>In addition to shared scopes, an entry may have an entry-local scope tha applies only to itself—allowing truly per-entry schema/settings/rules extensions.</p>

</Section>

<Section id="sec-3-1-2" title="ClickUp's Inadequacy">
      <p>ClickUp has one core object: a task. So no matter what you’re trying to model, every entry is forced to inherit
        the same base schema— fields like Assignees, Due Date, Priority, and Status.
        That’s fine when the entry is a work item, but it becomes a mismatch when the entry represents
        something else entirely, like a record or event. Those fields are either meaningless or create the wrong mental
        model, and while you can hide some columns in specific views, you can’t truly remove the task-shaped defaults
        across ClickUp.</p>
</Section>
</Section>
<Section id="sec-3-2" title="Custom Fields">
      <p>These are fields you create and attach to scopes. In practice, scopes include location scopes
        (Spaces/Folders/Lists) and also non-location scopes such as Task Types. A Custom Field becomes available
        wherever its scope applies.</p>
</Section>
<Section id="sec-3-3" title="Field Types">
      <p>ClickUp provides a range of field types that differ not only in the kind of value they hold, but also in how
        they are configured, computed, restricted, and used within the platform. This section examines those field
        types and highlights some of their main properties.</p>
<Section id="sec-3-3-1" title="System-Only Field Types">
      <p>Some field types exist only as part of ClickUp’s built-in system and aren’t available as Custom Field types you
        can create. Examples include Lists, Status, Task Type, Tags, Priority, Dependencies, and Time Estimate.</p>
</Section>
<Section id="sec-3-3-2" title="Parameterized Field Types">
      <p>Some field types aren’t “complete” until you define parameters such as an option set, a scale, a range, or a
        referenced target.
        Common parameter patterns include option sets (Dropdown, Labels), scales (Rating), ranges (Progress Manual), and
        relationship/selection configuration (Rollup).</p>
</Section>

<Section id="sec-3-4" title="Scopes">
      <p>These are the two scope hierarchies used in ClickUp—one that answers “which project context is this entry part
        of?”, and one that answers “what kind of entry is this meant to represent?”.</p>
<Section id="sec-3-4-1" title="Location Scope Hierarchy">
      <p>This is the <strong>project hierarchy</strong>: the chain of project containers an entry sits inside. A space
        usually represents a broad domain (like a product area or department), folders group related sub-projects inside
        that domain, and lists are often the concrete project boards/pipelines where work is tracked day-to-day. When an
        entry belongs to a specific list, it’s automatically part of the larger project contexts above it too—so the
        entry is simultaneously in the “sub-project” scope (the list) and in the broader “project” scopes (its folder
        and space). That’s why location scopes behave like inheritance: lower-level project contexts don’t replace
        higher-level ones; they narrow them.</p>

<Section id="sec-3-4-2" title="Task Type Scope Hierarchy">
      <p>This is the <strong>purpose hierarchy</strong> (even though ClickUp implements it as flat). A task type scope
        doesn’t say where the entry lives; it says what it is. It’s the semantic category of the
        entry—what kind of data it represents and what it’s intended to be used for. For example, two entries can live
        in the same project scope, but have different purposes (a bug vs. a feature request vs. a meeting note), and
        that difference is captured by task type. In ClickUp, choosing a task type selects exactly one purpose scope
        with no inheritance: it doesn’t automatically imply a more general “parent type” scope. It’s a single-purpose
        label that lets you treat entries differently based on meaning, not location.</p>

<Section id="sec-4" title="Workspace">
<p>The workspace is the concrete implementation documented in this wiki. It describes the principal
  structures, conventions, and operating patterns that define what the workspace contains and how it is
  intended to function in practice.</p>
<p>A central concern in its design is the attempt to impose, as far as possible, a meaningful type scope
  over the workspace, even though ClickUp does not natively support general scope hierarchies beyond its
  project hierarchy. As a result, part of the workspace's structure reflects deliberate conventions and
  modeling decisions intended to compensate for that limitation and to preserve clearer semantic
  distinctions between different kinds of entries.</p>
<p>This section does not attempt to document every individual ClickUp element, location, or workflow in
  exhaustive detail. Rather, it focuses on the main pillars of the workspace: its core structures, principal
  conventions, recurring patterns, and central organizational themes. Its purpose is not to prescribe the
  construction and maintenance of every project in full detail, but to provide a clear account of the
  workspace's overall design and governing logic.</p>
<Section id="sec-4-1" title="Type Scope">
<p>The Type Scope hierarchy is built on the idea that the <BaseFieldLink to="task-type">Task Type</BaseFieldLink> field functions as a scope selector field. In ClickUp, this is not a formally recognized scope mechanism in the same way that location is for the project scope, but in our workspace we deliberately treat it as one. That is, we recognize that type meaningfully determines applicability, behavior, and structure, and we organize the workspace accordingly even if ClickUp itself does not systematically model task type as a true scope hierarchy.</p>
<p>This means that task type is not just a label for classification. It acts as a contextual boundary that helps determine what belongs to an entry and what does not. We create fields, status groups, tags, templates, automations, views, and filters that are intended for a specific task type and should not be used across others indiscriminately. The Type Scope section therefore describes the type-based structure we impose on the workspace: a conceptual scope hierarchy that we enforce through design, convention, and configuration rather than through a native ClickUp scoping system.</p>
<Section id="sec-4-1-1" title="Task Types">
<p>The following subsections present all task types used in our workspace together with the schema associated with each of them.</p>
<Section id="sec-4-1-1-1" title="Task">
<p>The default actionable work-item entry type, representing a unit of work that you intend to do, track to
  completion, and manage through a workflow. A Task has an owner (or owners), a lifecycle state, and often a due
  horizon; you can plan, prioritize, and advance it while keeping context in description, comments, attachments,
  and relationships. As the most operational type—meant for execution rather than just capture—it relies on
  ClickUp's built-in workflow machinery such as Status, Assignees, and Priority.</p>

<Section id="sec-4-1-1-2" title="Event">
<p>An Event represents something that happens at a defined period in time—it has a start and an end, and it is
  about when it occurs and when the user should be made aware of it in advance. Events let you record and track
  occurrences (meetings, deadlines, trips, etc.) and control how far in advance they surface for notification or
  display. The workspace derives precise Start Time and End Time from the dates you set, derives <TaskTypeFieldLink type="event" to="status">Status</TaskTypeFieldLink> from
  those times, and computes a <TaskTypeFieldLink type="event" to="relevance-date">Relevance Date</TaskTypeFieldLink> so the event can be shown or notified ahead of
  time.</p>

<p>ClickUp's <TaskTypeFieldLink type="event" to="start-date">Start Date</TaskTypeFieldLink> and <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink> default to 4:00 AM on the given day when

  is unsuitable for events, so <TaskTypeFieldLink type="event" to="start-time">Start Time</TaskTypeFieldLink> and <TaskTypeFieldLink type="event" to="end-time">End Time</TaskTypeFieldLink> exist as auxiliary
  computational fields: the system populates them from <TaskTypeFieldLink type="event" to="start-date">Start Date</TaskTypeFieldLink> and <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink>
  using the rules below, treating any time other than the 4:00 AM default as a real time and using it as-is. The
  user works only with <TaskTypeFieldLink type="event" to="start-date">Start Date</TaskTypeFieldLink> and <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink>; <TaskTypeFieldLink type="event" to="start-time">Start Time</TaskTypeFieldLink> and
  <TaskTypeFieldLink type="event" to="end-time">End Time</TaskTypeFieldLink> are not meant to be edited directly.</p>

<Section id="sec-4-1-1-7" title="Parent Task">

<Section id="sec-4-2-1" title="Status Groups">
<p>Status Groups are a parameter of ClickUp's base-schema field <BaseFieldLink to="status">Status</BaseFieldLink> that is set at the Location level
  (i.e., per list/folder/space), designed to model the lifecycle of a particular kind of entry. While a status
  group often ends up feeling like it belongs to a task type, that association is indirect: it's usually true in
  practice, but not guaranteed by ClickUp. In our workspace it becomes especially tight because every list is
  strongly coupled with a task type, so the status group chosen at the location level typically functions as the
  workflow track for that type.</p>
<p>ClickUp forces every status to live inside one of its built-in subgroups—<StatusSubgroup>Not started</StatusSubgroup>, <StatusSubgroup>Active</StatusSubgroup>, <StatusSubgroup>Done</StatusSubgroup>, <StatusSubgroup>Closed</StatusSubgroup>—even
  when the entry isn't really a "task" in the real-world sense. We still use those subgroups because we have to,
  and we try to map them as sensibly as possible—though for non-task entries it can sometimes feel a bit crooked.
  Whenever we mention a status, we annotate it with its subgroup as a pill.</p>
<Section id="sec-4-2-1-1" title="Status Transitions and Triggers">
<p>A status change is a transition: moving an entry from one lifecycle state to another. In our
  workspace, transitions can be driven by four kinds of triggers:</p>
<ul>
  <li><span class="trigger-type">Manual</span> — The user explicitly changes the status based on judgment calls and intangible factors that the workspace can't reliably capture or analyze.</li>
  <li><span class="trigger-type">Internal Conditional</span> — Some change in the entry or elsewhere in the workspace triggers the transition.</li>
  <li><span class="trigger-type">External Conditional</span> — A real-world event outside the workspace happens, but it's still something that can be pinpointed and articulated specifically.</li>
  <li><span class="trigger-type">Chronological</span> — The passage of time triggers the change: a particular date-time boundary is reached.</li>
</ul>
</Section>
<Section id="sec-4-2-1-2" title="Task Status Group">
<p>Connected to the task type <TaskTypeLink to="sec-4-1-1-1">Task</TaskTypeLink>, this status group is exclusive to that task type. It
  represents the lifecycle of an actionable work item.</p>
<ul>
  <LinkTarget as="li" id="status-task-backlog"><Status type="task" status="backlog" showSubgroup /> — The task exists, but there is no intent to work on it yet.</LinkTarget>
  <LinkTarget as="li" id="status-task-to-do"><Status type="task" status="to-do" showSubgroup /> — The task is a candidate for execution.</LinkTarget>
  <LinkTarget as="li" id="status-task-in-progress"><Status type="task" status="in-progress" showSubgroup /> — Work has begun on this task.</LinkTarget>
  <LinkTarget as="li" id="status-task-canceled"><Status type="task" status="canceled" showSubgroup /> — The task will not be completed.</LinkTarget>
  <LinkTarget as="li" id="status-task-complete"><Status type="task" status="complete" showSubgroup /> — The task is finished and successfully completed.</LinkTarget>
</ul>
<Section id="sec-4-2-1-2-1" title="Task Transitions">
<ul>
  <li><Status type="task" status="backlog" /> → <Status type="task" status="to-do" />: <span class="trigger-type">Internal Conditional</span> — Triggered when at least one of <TaskTypeFieldLink type="task" to="start-date">Start Date</TaskTypeFieldLink> or <TaskTypeFieldLink type="task" to="due-date">Due Date</TaskTypeFieldLink> is set.</li>
  <li><Status type="task" status="to-do" /> → <Status type="task" status="in-progress" />: <span class="trigger-type">Manual</span> — The user declares that work has begun.</li>
  <li>(<Status type="task" status="backlog" />, <Status type="task" status="to-do" />, <Status type="task" status="in-progress" />) → <Status type="task" status="canceled" />: <span class="trigger-type">Manual</span> — The user decides the task will not be completed.</li>
  <li>(<Status type="task" status="backlog" />, <Status type="task" status="to-do" />, <Status type="task" status="in-progress" />) → <Status type="task" status="complete" />: <span class="trigger-type">Manual</span> — The user declares the task has been completed.</li>
  <li><Status type="task" status="in-progress" /> → <Status type="task" status="to-do" />: <span class="trigger-type">Manual</span> — Work is paused, typically because it's blocked, deprioritized, or waiting on something, but still intended to be resumed.</li>
  <li>(<Status type="task" status="to-do" />, <Status type="task" status="in-progress" />) → <Status type="task" status="backlog" />: <span class="trigger-type">Internal Conditional</span> — Triggered when <TaskTypeFieldLink type="task" to="start-date">Start Date</TaskTypeFieldLink> and <TaskTypeFieldLink type="task" to="due-date">Due Date</TaskTypeFieldLink> are reset to <strong>null</strong>.</li>
</ul>
</Section>

<Section id="sec-4-2-1-3" title="Event Status Group">
<p>Connected to the task type <strong>Event</strong>, this status group represents the lifecycle of an event.</p>
<ul>
  <LinkTarget as="li" id="status-event-not-scheduled"><Status type="event" status="not-scheduled" showSubgroup /> — The event exists, but no <TaskTypeFieldLink type="event" to="start-date">Start Date</TaskTypeFieldLink> and no <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink> are set.</LinkTarget>
  <LinkTarget as="li" id="status-event-upcoming"><Status type="event" status="upcoming" showSubgroup /> — <TaskTypeFieldLink type="event" to="start-date">Start Date</TaskTypeFieldLink> or <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink> is set.   <LinkTarget as="li" id="status-event-occurring"><Status type="event" status="occurring" showSubgroup /> — The current date-time is on/after the event's <TaskTypeFieldLink type="event" to="start-time">Start Time</TaskTypeFieldLink> (i.e., the event is happening now or has started and hasn't ended yet).</LinkTarget>
  <LinkTarget as="li" id="status-event-occurred"><Status type="event" status="occurred" showSubgroup /> — The current date-time is after the event's <TaskTypeFieldLink type="event" to="end-time">End Time</TaskTypeFieldLink> (operationally: after the <TaskTypeFieldLink type="event" to="due-date">Due Date</TaskTypeFieldLink>/Time, which we treat as the event's end).</LinkTarget>
</ul>
<Section id="sec-4-2-1-3-1" title="Event Transitions">
<ul>
  <li><Status type="event" status="not-scheduled" /> → <Status type="event" status="upcoming" />: <span class="trigger-type">Internal Conditional</span> — Triggered when at least one of Start Date or Due Date is set.</li>
  <li><Status type="event" status="upcoming" /> → <Status type="event" status="occurring" />: <span class="trigger-type">Chronological</span> — Triggered when the current datetime reaches or passes the event's start datetime.</li>
  <li><Status type="event" status="occurring" /> → <Status type="event" status="occurred" />: <span class="trigger-type">Chronological</span> — Triggered when the current datetime reaches or passes the event's end datetime (i.e., the Due/End Time).</li>
  <li>(<Status type="event" status="upcoming" />, <Status type="event" status="occurring" />) → <Status type="event" status="not-scheduled" />: <span class="trigger-type">Internal Conditional</span> — Triggered when Start Date and Due Date are reset to null.</li>
  <li><Status type="event" status="occurring" /> → <Status type="event" status="upcoming" />: <span class="trigger-type">Internal Conditional</span> — Triggered when the event's start datetime is moved forward to a datetime that has not yet happened.</li>
</ul>

<ul>

<Section id="sec-4-2-2" title="Tags">
